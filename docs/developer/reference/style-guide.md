# 写作与风格约定

`docs/developer/` 与插件开发者材料的文风合同。维护者文档、用户向 guide 另有口径；developer **禁止**亲民导读体。

## 产品与版本称呼

| 场景 | 写法 | 禁止 |
| --- | --- | --- |
| 产品名 | **Pallas-Bot** | 「Pallas 4.0」把版本当产品名 |
| 现行契约 / 架构正文 | 直接陈述现行事实；必要时写「当前主线」「现行」 | 每段重复「4.0」当口头禅 |
| 与历史对照 | 「相对 v3」「迁移自 3.x」 | 用「4.0」暗示一切页面都在讲发行营销 |
| 依赖 / 包版本 | semver 字面量，如 `pallas-core>=4.0.0,<5.0.0` | 把约束写成「Pallas 4.0」散文 |
| 发行说明 / 迁移专页 | 可用「Pallas-Bot 4.x」点一次大版本 | 开发契约页通篇刷版本号 |

对标习惯：NoneBot 正文写「NoneBot / NoneBot2」，具体版本放在文档站版本切换与 changelog，不在每页口号化。

## 读者

| 文档树 | 读者 |
| --- | --- |
| `docs/maintainer/` | 站长 / 运维 |
| `docs/developer/` | 平台与插件作者 |
| 插件 README | 站长安装与使用（可含简短开发链出） |

禁止在同一页混写三类视角的长叙事。

## Developer 正例 / 反例

| 禁止（亲民 / 教练） | 改为（工程契约） |
| --- | --- |
| 「这页帮你判断…」 | 一句话目的 + 表 |
| 「很实际的判断题」 | MUST / MUST NOT 或决策表 |
| 「你最该关心」 | 字段 / API / 路径表 |
| 「少走很多弯路」 | 禁止假设表 |
| 空洞「已经是当前事实」 | 旁注模块路径或类型名 |
| 标题塞 ticket 号 | ticket 放正文或删 |
| 「Pallas 4.0 / 在 4.0 下」口头禅 | 「Pallas-Bot」或省略版本 |

主语优先：系统、契约、API；少用第二人称说教。

## 页面骨架

1. 一句话目的  
2. 范围（In / Out）  
3. 合同表（字段 / 默认 / 副作用）  
4. 代码或路径锚点  
5. 禁止项  
6. 相关链接  

## 插件 README

至少：做什么、是否单独安装、口令、配置入口、排障。架构长文与运维细节不堆进同一页。
