# 写作与风格约定

这页统一 Pallas 文档、插件说明和开发者材料的口径，不是通用写作课。

## 基本原则

- 面向对象明确
- 不混写维护者与普通用户视角
- 文案短句优先
- 架构图配说明
- 运行时边界要写清楚

## 文档站层面的约束

当前文档站优先区分两类读者：

- 维护者
- 开发者

普通用户手册后续单开，不再和这两类内容混写。

## 插件 README 的写法

插件 README 至少交代：

- 这个插件做什么
- 是否需要单独安装
- 有没有口令
- 配置项在哪里
- 常见排障入口在哪

不要把架构设计、运维细节、用户口令和开发者实现混成一页。

## 架构文档的写法

新的架构文档应该遵循：

- 先讲现行事实，再讲原因
- 少写阶段路线图，多写稳定边界
- 一页只剩历史价值，就不该继续占主入口

## 说明代码时

- 优先给边界和职责
- 少做逐行翻译
- 路径、模块名、命令 ID 要写准确
