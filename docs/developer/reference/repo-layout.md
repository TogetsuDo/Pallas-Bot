# 仓库结构

这页帮你分清整个 4.0 体系里有哪些仓库、哪些边界是稳定的，而不只是“主仓里有哪些目录”。

## 先分清这些仓库

| 仓库 | 责任 |
| --- | --- |
| `Pallas-Bot` | 主仓，本体运行时、内核、内置插件、WebUI 后端、分片与治理能力 |
| `Pallas-Bot-WebUI` | WebUI 前端源码仓 |
| `Pallas-Bot-AI` | AI runtime 仓，负责媒体与 AI 任务执行 |
| 官方扩展仓库 | 独立能力包，例如 `duel`、`draw`、`maa`、`who-is-spy` |
| 社区扩展仓库 | 第三方生态与站点自定义能力 |

## 为什么这件事重要

4.0 里最常见的误判之一，就是在错误的仓库里找答案。

例如：

- 控制台页面问题，不一定在主仓前端代码里。
- AI 任务问题，不一定在 `Pallas-Bot` 主仓里修。
- 扩展命令或 README 问题，往往应在对应兄弟仓处理。

所以开发前先判断“这件事属于哪个仓”，比先动手改代码更重要。

## 主仓里最重要的目录

当前 4.0 主仓至少要认识这些目录：

| 路径 | 作用 |
| --- | --- |
| `pallas/` | 内核与平台实现 |
| `packages/` | 内置插件 |
| `tests/` | 回归与分层测试 |
| `docs/` | 维护者与开发者文档 |
| `config/` | 配置示例与部署模板 |
| `data/` | 运行时数据 |
| `local/plugins/` | 站点私有插件目录 |

## 这些目录分别意味着什么

### `pallas/`

这里是主仓真正的内核实现区。

如果你在改：

- 平台能力
- 配置读写
- 运行时装配
- WebUI 后端能力
- 分片协同

通常都会落到这里。

### `packages/`

这里放的是内置插件，不是所有外部扩展。更准确地说，它承载：

- 随主仓一起维护的插件
- 需要和主仓版本强同步的内置能力

### `local/plugins/`

这里更适合：

- 站点私有插件
- 临时试验型插件
- 不准备进入主仓或官方扩展仓的本地能力

### `data/`

::: warning 这是运行时目录，不是源码目录
很多问题最后能在这里找到证据，但别把它误当成可长期提交的源码区域。
:::

`data/` 里常见的证据包括：

- WebUI 落盘配置
- 分片 registry
- 运行日志
- 静态资源产物

## 这些目录通常由谁改

| 路径 | 常见修改者 |
| --- | --- |
| `pallas/` | 主仓维护者、平台能力开发者 |
| `packages/` | 内置插件维护者 |
| `tests/` | 所有功能开发者 |
| `docs/` | 维护者与开发者文档维护者 |
| `local/plugins/` | 站点维护者、私有插件作者 |

## 最容易混淆的四个边界

### WebUI

- 前端源码：`Pallas-Bot-WebUI`
- 主仓运行产物：`data/pb_webui/public/`

如果你在主仓里直接改 `data/pb_webui/public/`，你改到的是运行产物，不是前端源码。

### AI

- 产品与任务编排边界：主仓定义
- provider、执行、callback runtime：`Pallas-Bot-AI`

如果问题出在模型执行链或 AI 服务本体，不要默认在主仓里硬修。

### 官方扩展

很多 4.0 能力已经拆到独立仓库。

因此：

- 主仓里看不到该插件源码，不代表它不存在
- 某个玩法更新，未必应该进 `packages/`

### 文档

- `docs/maintainer/` 与 `docs/developer/` 是现行主线
- `docs/architecture/` 现在更多承担深度资料与迁移素材角色

不要再把旧时代目录说明当成正式导航结构。

## 开发前的判断顺序

拿到一个改动时，先判断：

1. 这是主仓问题，还是兄弟仓问题
2. 这是内核能力，还是插件能力
3. 这是源码问题，还是运行产物问题
4. 这是现行文档主线，还是历史资料

这个判断顺序能帮你少走很多弯路。
