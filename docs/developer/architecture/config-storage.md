# 配置与存储

这页讲 4.0 下配置事实源、读取方式和热重载边界该怎么设计，不是讲“去哪里改配置”。

## 先记住三个事实源

处理配置时，至少要分清这三个：

- `config/pallas.toml`
- 遗留 `.env` / `.env.{ENVIRONMENT}`
- `data/pallas_config/webui.json`

运行时不能只看其中任意一个文件。

## 这三个来源分别意味着什么

### `pallas.toml`

更偏启动层和基础设施层：

- 监听
- 数据库
- 部署前提
- 基础环境

如果你的能力依赖这些信息，通常是“进程如何启动”的一部分。

### 遗留 `.env`

4.0 下它的地位已经降级：

- 仍可被只读合并
- 主要为历史兼容和第三方插件保留
- 不该再作为新主能力的首选配置入口

### `webui.json`

这是运行态配置的核心落盘：

- 插件页配置
- 通用配置段
- WebUI 覆盖项

如果你的插件或平台能力支持运行中修改，最终通常都要和这里对接。

## 谁是最终事实源

比起“存在哪”，你更该关心“谁是最终事实源”。当前 4.0 合并顺序是：

1. `pallas.toml`
2. `.env`
3. `webui.json`

所以：

- 代码里不要默认 `pallas.toml` 就是最终值
- 也不要直接从环境变量读就认为已经拿到现行配置

稳妥的做法是统一走仓库配置读取入口。

## 你该怎么读配置

通过现有统一入口读取，别让各模块自己拼读取顺序。

也就是：

- 平台或基础能力优先走仓库配置读取 API
- 插件能力优先走现行插件配置接法
- 不要在业务代码里散落“先读环境、再读文件、再读 fallback”的私有实现

每个插件都自造一套读取逻辑，后续 WebUI 覆盖、热重载和分片一致性都会失控。

## 插件配置的推荐接法

在当前主线里，插件常见有两种接法：

### 插件页配置

适用于：

- 有明确插件配置页
- 需要通过 WebUI 调整
- 希望热重载或统一治理

### 通用配置段

适用于：

- 横切能力
- 多插件共享配置
- 不是某单一插件私有设置

如果一项配置已经明显是“平台级横切项”，就别再硬塞进某个插件私有配置页。

## 热重载怎么理解

::: warning 保存成功 ≠ 运行中已更新
开发者最常误判的就是这一点。配置保存成功，不代表运行中的进程已经更新。
:::

先分清：

- 这是启动层配置，还是运行态配置
- 这是普通参数，还是会触发运行时副作用
- 当前插件是否已接入 `install_hot_reload_config`

再实际一点说：

- 不是所有配置都该热重载
- 也不是所有能热重载的配置都该靠进程重启解决

## 分片下为什么更要统一读取入口

单进程里，一些随手写的配置读取方式也许还能凑合工作。

分片下就会立刻暴露问题：

- hub 和 worker 拿到的值不一致
- WebUI 已经保存，worker 却没感知
- 某个模块只看本进程环境，忽略了共享落盘

所以写分片相关功能时，默认把配置当成“跨进程一致性问题”，而不是单模块局部变量。

## 最常见的三类错误

### 直接读 `os.environ` 当最终值

这会绕过仓库合并顺序，也绕过 WebUI 覆盖。

### 在插件里自造私有配置文件

这会脱离当前治理和热重载体系，后续维护成本很高。

### 把平台横切配置塞进单插件私有页

短期看似方便，长期会让配置边界越来越混乱。

## 开发前的判断题

拿到一项新配置时，先问自己：

1. 这是启动层还是运行态配置
2. 这是单插件私有项还是平台横切项
3. 它是否应该支持 WebUI 修改
4. 它是否需要热重载
5. 分片下 worker 是否要读到同样的值

把这五个问题答清楚，配置设计通常就不会跑偏。
