# Core 与扩展

这页帮你判断：一个能力该放进 core，还是做成官方扩展或社区插件。

4.0 的方向很明确，判断标准也得明确，别靠“现在写起来放哪方便”来决定。

## 三层分工

| 层 | 放什么 |
| --- | --- |
| `Core` | 所有站点共用的平台能力、产品底盘、统一治理能力 |
| `Official Extensions` | 官方维护、但可以独立安装和发布的玩法与能力包 |
| `Community Extensions` | 第三方插件、站点定制、实验性功能 |

## 什么应该进 Core

满足下面任一条的能力，优先考虑进 core：

- 是所有部署形态都会依赖的运行时基础设施
- 是命令权限、帮助系统、配置热重载、分片协调这类平台共性
- 直接定义了 Pallas 的产品底盘，例如语料、ingress、多牛协同、核心人格相关边界
- 需要被大量插件复用，且不适合作为可选依赖存在

典型例子：

- 插件加载与治理
- WebUI 后端能力
- 命令权限与 cooldown
- 分片协调与 ingress 路由
- 核心帮助系统与基础状态能力

## 什么更适合做 Official Extension

下面这些通常不该再塞回 core：

- 独立玩法
- 外部服务集成
- 可以由站点按需安装的媒体能力
- 与某个垂直场景强绑定的功能包

典型例子：

- `duel`
- `who_is_spy`
- `maa`
- `draw`
- `dream`
- `ai_media`
- `protocol` 相关扩展能力包

做成官方扩展的好处很实际：

- 可以独立发版
- 可以独立 README 与 PyPI 包
- 主仓能保持更清晰的产品边界

## Community Extension 的定位

社区插件不是“次一级官方插件”，而是另一条演进轨道。

适合放进 community 的情况：

- 站点私有需求
- 还不确定是否进入官方生态
- 对外部系统、特定群体或特定玩法强耦合
- 维护者并不是主仓核心维护团队

## 一个实用判断题

加新功能前，先问四个问题：

1. 没有它，Pallas 作为平台还能不能正常成立？
2. 这个能力是不是所有站点都该默认拥有？
3. 它会不会显著增加 core 的发布、测试、治理负担？
4. 它是否天然适合独立安装、独立文档、独立发布？

第 1、2 题都偏“是”，通常更接近 core。
第 3、4 题偏“是”，通常更接近 extension。

## 不该做的两种回退

### 回退到“大一统 core”

把所有玩法、外部服务和实验功能都塞回主仓，会直接破坏 4.0 的边界。

### 回退到“每个插件自造平台能力”

插件不该各自重写一套权限、冷却、配置页、帮助文案、热重载、状态落盘逻辑。平台共性应该回到 core 提供统一入口。

## 与插件治理的关系

三层分工不是文档概念，而是治理概念：

- core 插件遵循更强的结构约束
- 官方扩展有统一安装、版本、治理入口
- 社区插件以索引、Git 仓或本地目录方式接入

所以设计新能力时，先决定归属层，再开始写代码。
