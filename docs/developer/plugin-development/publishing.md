# 发布

Pallas 4.0 下，“发布”不是一个动作，而是三条不同的分发路径。先分清你在发布哪类能力，再决定仓库、版本和交付方式。

## 先分清三条路径

1. 官方扩展 PyPI 发版
2. 社区插件公开收录与分发
3. 站点本地插件投放

这三条路径的目标、读者和约束都不同。

## 为什么要先分清路径

最常见的误区不是“不会发版”，而是：

- 明明是官方扩展，却按本地私有插件随手处理
- 明明是社区插件，却硬套官方 PyPI 发布节奏
- 明明只是站点私有能力，却过早设计完整公共发行体系

4.0 的发布设计和分层设计是一体的。能力属于哪一层，决定了它该怎么发布。

## 路径一：官方扩展

官方扩展的目标是：

- 可独立安装
- 可独立升级
- 有稳定版本号
- 能被站长清晰识别为正式能力包

### 这类仓库通常要负责什么

- `pyproject.toml` 版本
- README 徽章与安装说明
- tag 与版本对齐
- PyPI 发布
- 与主仓能力和治理方式保持协同

### 什么时候应该走这条

满足下面这些的插件，更接近官方扩展：

- 由官方维护
- 需要长期独立演进
- 希望站长像安装官方能力一样安装和升级
- 应该有清晰 semver 与包管理入口

### 开发者发布前最该确认什么

- 包名、仓库名、版本号是否一致
- README 是否说明安装、升级、依赖和适用范围
- metadata、命令权限、治理能力是否完整
- 是否已准备 PyPI 发布链路

### 这一条路径的核心交付物

- PyPI 包
- Git tag
- 对应版本 README
- 可被主仓或站长稳定安装的依赖入口

现行细节说明见：

- [官方扩展 PyPI 发版](../../develop/extension-pypi-publish.md)

## 路径二：社区插件

社区插件更关注的是“如何被发现、理解和安装”，而不一定要求走官方扩展那套发行体系。

### 常见分发方式

- 被索引仓收录
- 通过 Git 直装
- 被站长手工放到本地插件目录

### 这类发布更看重什么

- README 是否清楚
- metadata 是否完整
- 插件 ID、目录结构、图标是否规范
- 是否明确最低适配版本

### 什么时候应该走这条

满足下面这些的插件，更接近社区插件：

- 由第三方、社群或站点维护
- 不一定进入官方发行体系
- 更强调发现性和接入便利
- 主要目标是被生态收录和安装

### 开发者最常忽略的点

- 能跑起来，但 README 不足以让站长理解怎么装
- metadata 不完整，导致帮助、治理或索引展示不理想
- 目录和图标不规范，影响收录和商店呈现

现行细节说明见：

- [社区插件开发者指南](../../guide/community-plugin-author.md)

## 路径三：站点本地插件

这条路径不是公开发行，而是站点级投放。

适合：

- 私有能力
- 内网能力
- 实验性能力
- 明确不打算进入公共生态的插件

### 这条路径的重点不是“发版”，而是“可维护”

你仍应至少保证：

- 目录结构清楚
- README 能说明用途与依赖
- metadata 不要完全缺失
- 不要直接依赖主仓内部实现细节

它不一定对外发布，但结构过于随意，后续迁到社区插件或官方扩展会非常痛苦。

## 什么时候该选哪条

先问自己四个问题：

1. 这个能力是不是官方长期维护的一部分？
2. 它是否需要稳定版本号和独立升级？
3. 它的主要目标是公开被发现，还是站点内部使用？
4. 它是否值得为公共生态承担更严格的 README、图标、metadata 和发行责任？

1、2 偏“是”，通常更接近官方扩展。
3、4 偏“公开生态”，通常更接近社区插件。
主要是站点内部使用，通常更接近本地插件。

## 发布前最少检查

无论哪条路径，至少都应检查：

- README 是否说明用途、安装方式、依赖和版本范围
- metadata 是否已接入现行治理方式
- 图标、ID、目录结构是否清楚
- 最小测试是否已补

## 官方扩展发布前额外检查

- `pyproject.toml` 版本是否正确
- tag 是否与版本一致
- README 徽章是否同步
- PyPI 发布链路是否可用

## 社区插件发布前额外检查

- 索引收录信息是否完整
- Git 地址是否稳定可访问
- 图标和描述是否适合商店展示
- 最低 Pallas 版本是否写清楚

## 本地插件投放前额外检查

- 是否真的不需要公共发行
- 是否已避免对内部 API 的长期耦合
- 是否保留了未来迁移到社区插件的空间

## 最常见的三类错误

### 把“本地可用”误当成“已完成发布”

能在自己站点跑，不代表别人能安装、理解和维护。

### 只做版本号，不做生态交付物

真正的发布不只是 bump 版本，还包括 README、徽章、tag、索引或包分发入口。

### 路径选错

一开始就把路径选错，后面版本管理、仓库拆分和文档结构都会越来越乱。

## 和架构分层的关系

发布路径不是独立话题，它直接对应 4.0 的三层能力归属：

- `Core` 不按插件发行
- Official Extension 走正式独立发布
- Community Extension 走生态分发
- 站点私有能力优先走本地插件路径

所以更稳妥的顺序通常是：

1. 先判断归属层
2. 再决定发布路径
3. 最后再设计版本和交付物
