# 命令权限

这页帮你理清命令权限。它的难点不是「系统不会用」，而是分不清三个值：代码默认值、WebUI 覆盖值、帮助展示值。

## 先记住三件事

1. 插件代码里声明的是默认权限。
2. WebUI 可以覆盖运行时权限。
3. 帮助里的「何人可用」应展示当前生效值，而不是写死在文案里。

## 你平时在管什么

命令权限主要解决：

- 默认谁能用
- 运行中改成谁能用
- 帮助页怎么同步展示

它不负责：

- 命令 cooldown
- 插件是否安装
- 插件是否全局禁用

这些属于其他治理面。

## 怎么理解当前的权限等级

你通常会看到这几个等级：

- `everyone`
- `bot_moderator`
- `group_moderator`
- `staff`
- `superuser`

关键是记住：它们是运行时判定等级，不是帮助文案标签。

## 权限变更的优先级

按这个顺序理解：

1. 插件代码声明默认值
2. 配置或环境变量提供覆盖
3. WebUI 保存后的覆盖值最高

::: warning 别只看源码默认值
现场排障时，插件 README 或源码默认值都不算数。要确认的是当前实例最终生效的覆盖值。
:::

## 你最常做的三件操作

### 1. 看默认权限是否合理

适用于：

- 新插件接入
- 升级后帮助展示变化
- 某命令默认开放范围不合理

### 2. 在 WebUI 调整运行时覆盖

适用于：

- 临时提高或降低某条命令的使用门槛
- 不改代码先做运维层收口

这类变更通常不需要重启。重点是确认保存后覆盖值已经落盘并清缓存生效。

### 3. 核对帮助展示与真实鉴权是否一致

如果用户反馈「帮助说能用，但实际不能用」，优先怀疑：

- 帮助绑定的命令 ID 不对
- 鉴权使用的命令 ID 不对
- WebUI 覆盖值没有被正确纳入展示

## 排障顺序

### 某命令突然谁都不能用

先查：

- WebUI 是否把权限调高了
- 该命令是否还有其他业务前提
- 插件是否被全局禁用

### 帮助里展示和实际权限不一致

先查：

- 命令 ID 是否一致
- 插件 metadata 是否声明了 `command_permissions`
- 当前帮助图是否拿到了最新治理数据

### WebUI 里没有这条命令

通常说明：

- 插件 metadata 没声明该命令
- 或 registry 底表里没有它

别急着当成 WebUI bug，先回头看插件接入是否完整。

## 和 cooldown 的区别

很容易把「不能立刻再用一次」误当成权限问题。分清这两者：

- 命令权限：决定谁能用
- cooldown：决定多久能再用一次

第一次用就被拒绝，先查权限。刚用过又失败，才去查 cooldown。

## 使用原则

- 先看当前生效值，不要只看源码默认值。
- 先区分权限、禁用、cooldown，再判断问题在哪一层。
- 不要把帮助文案本身当成唯一事实来源。
