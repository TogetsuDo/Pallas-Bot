# WebUI 运维

这页帮你判断运行中的控制台是否正常，以及出问题时该先看哪一层。

## 先分清三层

WebUI 的问题，至少要分清这三层：

- 前端源码仓：`Pallas-Bot-WebUI`
- 主仓运行产物：`data/pb_webui/public/`
- 主仓 WebUI 后端：`pb_webui` 和 `/pallas/api`

不同层出问题，现象很像，处理方式却完全不同。

## 运行时先看什么

### 页面打不开

优先检查：

- Bot 是否正常启动
- `pb_webui` 是否正常挂载
- 路径是否为 `/pallas/`

### 页面能打开，但内容像旧版

优先检查：

- `data/pb_webui/public/` 是否是新产物
- 浏览器缓存是否强制刷新
- 是否只改了源码仓但没同步运行产物

### 页面能打开，但保存失败

优先检查：

- `/pallas/api` 是否可达
- 写接口是否返回鉴权或 500
- 对应插件是否支持当前配置保存

## 哪些页面依赖插件能力

不是所有控制台页面都只依赖 `pb_webui`。比如：

- 协议端相关页面依赖协议端插件
- 插件页依赖插件 metadata 和能力聚合
- AI 相关页面依赖 AI 运行态接口与聚合逻辑

所以「页面空了」不一定是前端资源问题，也可能是后端插件状态出了岔子。

## 什么时候该去看前端仓

只有这些场景，才优先去 `Pallas-Bot-WebUI`：

- 布局错乱
- 组件没渲染
- 交互按钮位置或样式不对
- 移动端或窄屏适配问题

## 什么时候该优先看主仓

- API 返回异常
- 保存配置失败
- 页面依赖的运行态数据不对
- 插件列表、权限、治理信息不完整

## 建议的检查顺序

1. 页面路径与登录是否正常
2. 运行产物是否同步
3. `/pallas/api/health` 与基础接口是否正常
4. 对应后端插件是否加载
5. 再区分前端问题还是后端问题
