# 插件治理

插件治理不是一个按钮，而是一组运行时能力。抓住「安装、可见、可用、生效方式」四个维度就够了。

## 先看哪四件事

治理插件时，先判断：

1. 插件是否已安装
2. 插件是否对当前环境可见
3. 插件是否真的能被使用
4. 插件改动后怎样生效

这四件事不是一回事。很多「插件没生效」的问题，本质上只是其中一层出了偏差。

## 你平时在管什么

- 帮助可见性
- 全局禁用
- 单插件治理页
- 命令权限覆盖
- cooldown 展示与覆盖
- 插件配置页
- `reload_policy`
- `activation_policy`

## 这几个概念分别管什么

### 帮助可见性

决定插件是否在帮助体系里展示。

它影响“能不能被看到”，不等于插件是否真的被禁用。

### 全局禁用

决定插件是否在当前实例整体不可用。

它影响“能不能运行”，而不只是帮助页里是否展示。

### 命令权限

决定哪些角色能触发某条命令。

它影响“谁能用”，不等于插件整体可用性。

### cooldown

决定某些命令的频率限制。

它影响“多久能再用一次”，不是权限系统。

### `reload_policy`

决定配置、元数据或代码类变更如何生效。可以理解成「改完要不要重启、重载到哪一层」。

### `activation_policy`

决定插件安装、更新、启停后，运行时该做什么激活动作。常见三种：

- 可直接热加载
- 需要 worker 重启
- 需要整套重启

## 三个最常用的治理入口

### 1. 插件列表

适合先看：

- 插件是否存在
- metadata 是否完整
- 基本状态是否符合预期

### 2. 单插件治理页

适合看：

- 指令与能力摘要
- 当前禁用 / 可见状态
- perm / cooldown 覆盖
- `activation_policy`

### 3. 通用配置与权限矩阵

适合看：

- 跨插件统一权限覆盖
- 全局行为类配置
- WebUI 覆盖与代码默认值是否对不上

## 排障时怎么判断

### 安装了，但帮助里看不到

先查：

- 帮助可见性
- 插件 metadata
- 插件是否被全局禁用

### 帮助里能看到，但实际不能用

先查：

- 命令权限覆盖
- cooldown
- 插件是否仍被禁用
- 当前部署是否真的加载了该插件

### 配置改了，但行为不变

先查：

- 插件是否接了 WebUI 配置
- `reload_policy` 是否支持当前变更直接生效
- 当前改的是配置、元数据还是代码

### 分片下 WebUI 显示不对

先查：

- hub 聚合接口
- worker metadata / capabilities 是否正常汇总
- 是否误把 hub 本地状态当成全局状态

## 基本原则

- 不要把「帮助可见」误当成「插件可用」。
- 不要把「插件安装成功」误当成「运行态已经生效」。
- 分片下优先把治理数据视为 hub 聚合结果，而不是单进程本地状态。

## 配合查看的 API

这些接口域可以直接看：

- `/pallas/api/plugins`
- `/pallas/api/plugins/capabilities`
- `/pallas/api/plugins/{plugin_name}/governance`
- `/pallas/api/plugins/help-menu-visibility`
- `/pallas/api/plugins/global-disable`
